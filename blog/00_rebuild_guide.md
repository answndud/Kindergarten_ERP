# Blog Rebuild Guide

이 문서는 `blog/` 시리즈를 **설명형 글**에서 **재현형 구현 가이드**로 강화하기 위한 공통 규칙 SSOT입니다.

## 1. 목표

목표는 단순합니다.

- 독자가 글을 읽고 설계를 이해하는 것에서 끝나지 않고
- 직접 손으로 따라 만들고
- 각 글이 끝날 때 체크포인트를 통과하게 만드는 것

즉, 앞으로 이 시리즈는 아래 두 역할을 동시에 가져야 합니다.

1. 설계 / 포트폴리오 설명
2. 재현 가능한 구현 가이드

## 2. 각 글에 반드시 들어갈 재현형 섹션

### A. 시작 상태
- 이전 글 기준 무엇이 준비되어 있어야 하는가
- 어떤 파일과 환경이 있어야 하는가

### B. 이번 글에서 바뀌는 파일
- 새 파일
- 수정 파일
- 삭제 파일

### C. 구현 체크리스트
- 독자가 직접 해야 할 작업을 번호 순서로 정리

### D. 실행 / 검증 명령
- 복붙 가능한 명령
- 가능하면 아래 두 층으로 나눈다
  - 단계 체크포인트: `blog/scripts/checkpoint-XX.sh`
  - 현재 저장소 기준 안정 검증: `compileJava compileTestJava`, `integrationTest`, `performanceSmokeTest`
- 성공 시 무엇이 보여야 하는지까지 포함

### E. 산출물 체크리스트
- 이 글이 끝났을 때 반드시 생겨 있어야 하는 파일
- 추가된 migration / endpoint / 화면 / 테스트 클래스
- 독자가 자기 저장소와 바로 대조할 수 있는 결과물

### F. 글 종료 체크포인트
- 이 글을 마치면 프로젝트가 어떤 상태여야 하는가

### G. 자주 막히는 지점
- 증상
- 가능한 원인
- 가장 먼저 확인할 것

## 3. 재현성 기준

각 글이 아래를 만족해야 “재현형 보강 완료”로 봅니다.

1. 최소 1개 이상의 실행 명령이 있다
2. 최소 1개 이상의 검증 명령이 있다
3. 파일 생성/수정 목록이 있다
4. 산출물 체크리스트가 있다
5. 글 종료 체크포인트가 있다
6. 입문자가 자주 실패하는 포인트가 있다

## 4. 검증 전략

이 저장소는 이미 최종 형태까지 성장한 프로젝트라서, 블로그 중간 단계를 설명할 때도 실제 저장소에는 후반 기능이 함께 들어 있습니다.
그래서 블로그 검증은 아래 두 층으로 나눕니다.

1. 단계 체크포인트 검증
   - `blog/scripts/checkpoint-XX.sh`
   - 파일/메서드/문서/마이그레이션이 해당 단계 산출물대로 있는지 확인
2. 현재 저장소 기준 안정 검증
   - `./gradlew compileJava compileTestJava`
   - `./gradlew --no-daemon integrationTest`
   - `./gradlew --no-daemon performanceSmokeTest`

좁힌 `--tests` 실행은 일부 환경에서 Gradle XML result writer 충돌이 날 수 있으므로, 블로그 SSOT는 `checkpoint script + 안정 스위트` 조합으로 둡니다.

## 5. 권장 읽기 순서

1. `00_series_plan.md`
2. `00_rebuild_guide.md`
3. 각 글 본문
4. 글 끝의 산출물 체크리스트와 검증 명령
5. 필요하면 `blog/scripts/checkpoint-XX.sh`

## 6. 우선 보강 대상

1. `02`~`05`
2. `11`~`15`
3. `16`~`21`
4. `22`~`26`
5. `01`, `06`~`10`

이 순서를 택한 이유는 환경 / 설정 / 인증 / 테스트를 먼저 재현 가능하게 해야 나머지 글도 따라갈 수 있기 때문입니다.
