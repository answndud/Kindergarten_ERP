# Blog Quality Checklist

이 문서는 `blog/` 시리즈를 고칠 때 사용하는 **실전 편집 체크리스트**입니다.

목표는 두 가지입니다.

1. 초보자가 글을 읽고 실제로 구현 순서를 이해하게 만들기
2. 취업 준비생이 글을 읽고 면접에서 바로 설명할 수 있게 만들기

이 문서는 “글을 더 예쁘게 쓰자”가 아니라, **각 글이 실제로 통과해야 하는 품질 기준**을 정리합니다.

## 1. 사용 방법

각 글을 수정할 때 아래 순서로 확인합니다.

1. `한 문장 요약`을 먼저 적습니다.
2. 아래 A~H 체크리스트를 위에서 아래로 검사합니다.
3. `예 / 아니오`가 섞여 있으면, `아니오` 항목부터 고칩니다.
4. 마지막에 `초보자 기준`, `면접 기준`, `현재 코드 기준`이 동시에 맞는지 다시 봅니다.

## 2. A. 초보자 이해도

### 반드시 Yes여야 하는 질문

- 글의 첫 10줄 안에 `이번 글이 풀 문제`가 구체적으로 보이는가?
- 이 글이 시리즈에서 왜 지금 나오는지 설명하는가?
- 처음 등장하는 용어를 바로 풀어서 설명하는가?
- 독자가 “무엇을 만들고 있는지”를 코드 전에 이해할 수 있는가?
- 정책이 복잡한 경우, 글 속에 `상황별 표`나 `before/after 비교`가 있는가?

### 특히 손봐야 하는 신호

- `lifecycle`, `rotation`, `aggregate`, `immutability`, `orchestration` 같은 단어가 정의 없이 바로 나온다
- “중요하다”, “안전하다”만 있고 왜 그런지 구체 사례가 없다
- 한 문단에 개념이 3개 이상 섞여 있다

### 권장 보강 방식

- 용어 첫 등장 시 `쉬운 말 -> 영어/전문 용어` 순서로 쓴다
- 정책 글에는 `상황 / 허용 여부 / 이유 / 사용자에게 보이는 결과` 표를 넣는다
- 흐름 글에는 `이전 상태 -> 현재 상태 -> 다음 글` 한 줄을 넣는다

## 3. B. 재현성

### 반드시 Yes여야 하는 질문

- `시작 상태`가 있는가?
- `이번 글에서 바뀌는 파일`이 있는가?
- `구현 체크리스트`가 번호 순서로 있는가?
- `실행 / 검증 명령`이 있는가?
- `산출물 체크리스트`가 있는가?
- `자주 막히는 지점`이 있는가?

### 특히 손봐야 하는 신호

- 글 안에서 없는 템플릿, 없는 스크립트, 없는 테스트 클래스를 안내한다
- 현재 저장소에서 실행되지 않는 명령을 그대로 적어 둔다
- “설명할 수 있다”만 있고, 어떤 파일/API/테이블이 생겨야 하는지 안 보인다

### 권장 보강 방식

- 체크포인트 스크립트가 실제로 없으면 문서에서 필수처럼 소개하지 않는다
- 실행 명령은 `현재 저장소 기준 실제로 도는 명령`만 남긴다
- 각 글 끝에 최소 아래를 적는다
  - 생성/수정 파일
  - 추가된 endpoint
  - 추가된 migration
  - 대표 테스트 클래스

## 4. C. 코드-문서 정합성

### 반드시 Yes여야 하는 질문

- 글에서 언급하는 클래스/메서드/파일이 현재 저장소에 실제로 존재하는가?
- 글의 설명이 실제 코드보다 과장되지 않았는가?
- 글의 URL, 계정, profile, 명령이 현재 코드와 일치하는가?
- 결정 로그/포트폴리오 문서 링크가 실제 파일과 맞는가?

### 특히 손봐야 하는 신호

- “안전하다”, “운영 가능하다”, “중복 없이 처리한다” 같은 강한 표현이 코드 근거 없이 나온다
- 실제 구현 한계가 있는데 글에서는 완성형처럼 말한다
- demo 계정, URL, API 경로가 코드와 다르다

### 권장 보강 방식

- 강한 표현 뒤에는 반드시 `근거 파일` 또는 `테스트`를 붙인다
- 구현 한계가 있으면 `현재 구현의 한계` 박스를 넣는다
- 시연 문서는 실제 `DataLoader`, controller mapping, security rule과 교차 검토한다

## 5. D. 면접 대응력

### 반드시 Yes여야 하는 질문

- `취업 포인트`가 있는가?
- 면접에서 바로 말할 수 있는 1~3개 답변 문장이 있는가?
- 꼬리 질문이 예상되는 지점을 짚는가?
- “왜 이 설계를 택했는가 / 대안은 무엇이었는가”까지 설명 가능한가?

### 특히 손봐야 하는 신호

- “무엇을 만들었다”만 있고 “왜 그렇게 했는가”가 없다
- 면접 답변이 기술 나열로 끝난다
- 트레이드오프가 안 보인다

### 권장 보강 방식

- 각 글 끝에 아래 3가지를 추가한다
  - 1문장 답변
  - 30초 답변
  - 예상 꼬리 질문 3개

## 6. E. 연재 흐름

### 반드시 Yes여야 하는 질문

- 이 글이 앞 글과 자연스럽게 이어지는가?
- 아직 설명하지 않은 후반 구현물을 너무 많이 끌어오지 않는가?
- 이 글의 범위와 다음 글의 범위가 겹치지 않는가?

### 특히 손봐야 하는 신호

- 도입부에서 후반부의 복잡한 기능을 한꺼번에 미리 공개한다
- 같은 주제를 두 글에서 중복 설명한다
- “나중에 고칠 예정”과 “지금 실제 구현”이 섞여 있다

### 권장 보강 방식

- 1편은 도메인 선택과 설계 이유에 집중하고, 후반 기능은 미리보기로만 짧게 둔다
- 한 글의 핵심 질문은 1개로 제한한다
- 다음 글과 역할이 겹치면 `이 글은 뼈대`, `다음 글은 고도화`로 구분한다

## 7. F. 문체와 용어

### 반드시 Yes여야 하는 질문

- 내부 문서 톤보다 공개 블로그 톤에 가깝게 쓰였는가?
- `SSOT`, `현재 저장소`, `오케스트레이터` 같은 표현이 남발되지 않는가?
- 영어 용어를 그대로 쓰더라도 문맥상 이해 가능한가?

### 특히 손봐야 하는 신호

- 문장마다 “즉,”가 반복된다
- 추상적인 명사형 표현이 길게 이어진다
- 구현 설명보다 내부 프로젝트 회고 톤이 강하다

### 권장 보강 방식

- 처음 한 번은 쉬운 한국어로 풀고, 그 다음 전문 용어를 붙인다
- 너무 내부자 같은 표현은 공개용 표현으로 바꾼다
- 한 문단은 한 메시지만 담는다

## 8. G. 현재 구현의 한계 고지

초보자용 글일수록, 그리고 면접용 글일수록 아래 질문을 꼭 확인합니다.

- 이 글이 설명하는 구조에 현재 코드 한계가 있는가?
- 있다면 독자가 오해하지 않게 적었는가?

### 권장 형식

```md
> 현재 구현의 한계
> 이 구조는 운영형으로 확장한 버전이지만, 아직 A/B는 남아 있습니다.
> 면접에서는 이 지점을 어떻게 개선할지까지 설명하면 더 좋습니다.
```

## 9. H. 글별 완료 기준

글 하나는 아래를 모두 만족해야 완료로 봅니다.

- 초보자 입장에서 “왜 이 글이 필요한지” 이해된다
- 현재 저장소 기준으로 실행/검증 경로가 거짓이 아니다
- 구현 산출물을 글만 보고 대조할 수 있다
- 면접 답변의 핵심 문장이 있다
- 실제 코드보다 더 강하게 주장하지 않는다

## 10. 우선 적용 순서

가장 먼저 손볼 글은 아래 순서가 좋습니다.

1. `11`, `16`, `18`, `21`
   - 보안/인증/Outbox는 용어 장벽과 과장 위험이 가장 큽니다.
2. `01`, `10`, `22`, `23`, `26`
   - 연재 흐름, 상태 전이 설명, 데모/면접 연결을 정리해야 합니다.
3. `02`~`05`
   - 초반 진입 장벽을 더 낮추고, 검증 경로를 단순화해야 합니다.
4. 나머지 글
   - 산출물 체크리스트와 꼬리 질문을 같은 형식으로 맞춥니다.

## 11. 빠른 점수표

글 하나를 빠르게 평가할 때는 아래처럼 봅니다.

- 초보자 이해도: `0~2`
- 재현성: `0~2`
- 코드-문서 정합성: `0~2`
- 면접 대응력: `0~2`
- 흐름/문체: `0~2`

총점 가이드:

- `9~10`: 공개해도 좋음
- `7~8`: 좋지만 보강 필요
- `5~6`: 읽히긴 하나 초보자/면접용으로는 부족
- `0~4`: 구조 재작성 필요
