# 목록 로딩 최적화 (HTMX + SSR)

## 문제
- 전체 페이지 리로드로 목록 갱신 시 체감 대기 시간이 길었다.
- 하드코딩된 목록/필터 값이 유지보수와 성능 이슈를 유발했다.

## 개선
- HTMX로 목록 영역만 부분 갱신하도록 전환.
- 필터/탭 변경 시 필요한 구간만 재요청하도록 구성.

## 효과(정성)
- 사용자 체감 로딩 시간이 줄고, 스크롤/필터 전환이 부드러워짐.
- 서버/네트워크 비용이 페이지 단위에서 컴포넌트 단위로 축소됨.

## 선택 이유
- SSR 환경에서 HTMX는 프론트 프레임워크 대비 구현 복잡도가 낮다.
- 페이지 유지/SEO/접근성 측면에서 SSR 강점을 유지하면서 성능 개선 가능.

## 예상 질문/답변
1) Q: SPA로 전환하지 않고 HTMX를 선택한 이유는?
   A: 현재 팀/시스템의 SSR 기반 템플릿 구조를 유지하면서도 부분 갱신으로 성능 개선이 가능했기 때문입니다.
2) Q: HTMX가 확장성에 문제되지 않나요?
   A: 현재 규모에서는 충분하며, 필요 시 API 기반 프론트로 확장할 수 있는 구조로 유지합니다.
3) Q: 목록 API를 그대로 사용해도 되나요?
   A: 페이지/필터 단위 호출로 최소한의 데이터만 가져오도록 설계했습니다.
4) Q: 사용자 피드백이 있었나요?
   A: 목록 갱신 시 전체 새로고침이 사라져 체감 지연이 감소했습니다.
5) Q: 성능 측정은 어떻게 했나요?
   A: 응답 시간과 체감 지연 감소를 중심으로 정성 평가했고, 추후 APM을 추가할 계획입니다.
