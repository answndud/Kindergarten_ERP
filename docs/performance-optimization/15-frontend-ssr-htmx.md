# 프론트엔드 선택: SSR + HTMX + Alpine.js

## 문제
- SPA 도입은 초기 설정/운영 복잡도가 높고, 팀 학습 비용이 크다.
- 템플릿 기반 시스템에서 페이지 전체 리렌더링이 잦았다.

## 개선
- Thymeleaf SSR로 구조를 유지하면서, HTMX로 부분 갱신 적용.
- Alpine.js로 간단한 상태/인터랙션만 처리.

## 효과(정성)
- 초기 렌더링이 빠르고, 필요한 부분만 갱신되어 체감 성능이 개선.
- 프론트 구조가 단순해 유지보수가 쉬움.

## 선택 이유
- SSR은 SEO/접근성/캐시 측면에서 유리.
- HTMX는 템플릿 구조와 잘 맞고, 프론트 전환 비용이 낮다.
- Alpine.js는 최소한의 JS로 상호작용을 제공한다.

## 예상 질문/답변
1) Q: SPA가 더 빠르지 않나요?
   A: 현재 규모에서는 SSR+부분 갱신이 충분하며 초기 로딩과 유지보수 비용이 낮습니다.
2) Q: HTMX의 한계는?
   A: 대규모 상태 관리에는 부적합하지만 현재 범위에는 최적입니다.
3) Q: Alpine.js를 선택한 이유는?
   A: 간단한 상태와 UI 토글에 최적화되어 있습니다.
4) Q: SEO가 중요한가요?
   A: 공지/알림장 페이지 등 정보 노출 측면에서 SSR의 이점이 있습니다.
5) Q: 추후 전환 계획은?
   A: 필요 시 API 중심 구조로 점진적 전환이 가능합니다.
