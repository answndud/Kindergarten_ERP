# 2026-02-14 성능 모의면접 플레이북

이 문서는 실제 면접 전 20~30분 동안 혼자 또는 스터디에서 바로 사용할 수 있는
"실전형 모의면접" 진행 가이드입니다.

## 1) 진행 방식 (권장 25분)

- 1분: 20초 자기소개 + 40초 성능 사례 답변
- 10분: 기본 질문 5개 (기술 설명형)
- 10분: 압박 질문 5개 (반박형)
- 4분: 피드백 반영 후 재답변

## 2) 준비물

- 기본 스크립트: `docs/00_project/2026-02-14-performance-story-script.md`
- 일반 Q&A: `docs/00_project/2026-02-14-performance-interview-qa.md`
- 압박 Q&A: `docs/00_project/2026-02-14-performance-pressure-qa.md`

## 3) 면접관 역할 질문 세트

### A. 기본 질문 (설명형)

1. 성능 병목을 어떻게 찾았나요?
2. Notepad N+1 개선 과정을 설명해 주세요.
3. Dashboard 통계 최적화에서 핵심은 무엇이었나요?
4. Redis refresh token 경로를 왜 바꿨나요?
5. 인덱스 튜닝 기준과 검증 방법은 무엇인가요?

### B. 압박 질문 (반박형)

1. H2 수치가 왜 신뢰 가능한가요?
2. 개선 폭이 작아 보이는데 과장 아닌가요?
3. 캐시를 먼저 쓰면 더 쉽지 않나요?
4. 인덱스 늘리면 쓰기 성능은요?
5. 다음 단계가 실제로 가능한 계획인가요?

## 4) 답변 점검 체크리스트

- 숫자를 먼저 말했는가? (예: 22 -> 4)
- 원인을 한 줄로 설명했는가? (예: readCount N+1)
- 해결 방법이 구체적인가? (집계 쿼리/인덱스/실행계획)
- 검증 근거를 제시했는가? (테스트 로그/EXPLAIN)
- 트레이드오프를 인정했는가?

## 5) 자주 하는 실수

- "빨라졌습니다"만 반복하고 수치를 말하지 않음
- 캐시, 인덱스, 쿼리 개선 순서를 설명하지 못함
- 왜 이 작업부터 했는지 우선순위 근거가 없음
- 한계/리스크 질문에 방어적으로만 답함

## 6) 마무리 한 문장 템플릿

"저는 기능 구현 이후 성능 병목을 수치로 검증하고,
조회 경로와 실행계획을 함께 개선해 재현 가능한 성능 향상을 만드는 방식으로 개발합니다."
