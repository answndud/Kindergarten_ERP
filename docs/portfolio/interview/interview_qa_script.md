# 면접 예상 질문/답변 스크립트

## 사용법

- 이 문서는 "실제로 면접에서 어떻게 말할지"를 바로 꺼내 쓰기 위한 스크립트입니다.
- 각 질문은 `짧은 답변`과 `꼬리 질문 대응 포인트`로 나눴습니다.
- 먼저 짧게 답하고, 면접관이 더 파고들면 아래 포인트를 확장해서 말하면 됩니다.

---

## 1. 프로젝트 소개

### Q1. 이 프로젝트를 한 줄로 소개해 주세요.

짧은 답변:
유치원 ERP를 만드는 과정에서, 기능 구현보다 멀티테넌시 권한 경계, 인증 세션 설계, 실환경형 테스트, 운영 관측성까지 보강한 백엔드 포트폴리오 프로젝트입니다.

꼬리 질문 대응 포인트:
- 단순 CRUD보다 운영 가능한 백엔드로 바꾸는 데 초점을 뒀다고 설명
- 권한, 보안, 테스트, 성능, 관측성을 순서대로 고도화했다고 설명

### Q2. 왜 유치원 ERP를 주제로 잡았나요?

짧은 답변:
역할이 원장, 교사, 학부모로 명확히 나뉘어서 권한 모델과 데이터 경계를 설계하기 좋았고, 공지, 알림장, 출석, 신청/승인처럼 백엔드에서 다뤄볼 수 있는 주제가 풍부했기 때문입니다.

꼬리 질문 대응 포인트:
- 단순 게시판보다 도메인 경계가 복잡해서 면접용 포인트가 많았다고 설명
- 멀티테넌시, 승인 워크플로우, 학부모-원생 관계 같은 실무형 문제를 담을 수 있었다고 설명

### Q3. 이 프로젝트에서 가장 강조하고 싶은 것은 무엇인가요?

짧은 답변:
기능 수보다, 기능이 많아질수록 생기는 보안 경계와 운영 리스크를 실제로 줄였다는 점입니다.

꼬리 질문 대응 포인트:
- 멀티테넌시 권한 하드닝
- JWT refresh/session 설계
- Testcontainers 기반 통합 테스트
- 감사 로그와 운영 관측성

## 2. 권한/보안

### Q4. 가장 큰 보안 문제는 무엇이었고 어떻게 고쳤나요?

짧은 답변:
초기에는 여러 API가 `id`만 알면 다른 유치원 데이터에 접근 가능한 IDOR 성격 문제를 갖고 있었고, 이를 requester 기반 서비스 검증과 공통 접근 정책으로 고쳤습니다.

꼬리 질문 대응 포인트:
- 컨트롤러에서 받은 `id`를 바로 신뢰하지 않도록 바꿨다고 설명
- 서비스 계층에서 같은 유치원, 본인 소유, 학부모-자녀 관계를 검증한다고 설명
- "기능 추가보다 데이터 경계가 먼저"라는 관점으로 재설계했다고 설명

### Q5. 권한 검사를 Controller가 아니라 Service에서도 한 이유는 무엇인가요?

짧은 답변:
컨트롤러만 믿으면 다른 진입점이나 내부 호출에서 쉽게 빠질 수 있어서, 비즈니스 규칙에 가까운 권한 검사는 서비스 계층이 최종 방어선이 되어야 한다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- 메서드 보안과 서비스 검증은 역할이 다르다고 설명
- URL 접근 제어는 1차, 데이터 경계 검증은 서비스가 2차라고 설명

### Q6. 감사 로그는 왜 FK 없이 저장했나요?

짧은 답변:
감사 로그는 운영 증적이라 회원 row의 lifecycle과 독립적으로 남아야 하기 때문입니다.

꼬리 질문 대응 포인트:
- 회원 탈퇴 후에도 로그인/소셜 연결 이력은 살아 있어야 함
- FK를 걸면 보존 정책과 충돌할 수 있음
- 대신 write-time에 `kindergartenId`를 함께 기록하고, tenant 귀속이 불가능한 경우만 `null`로 둔다고 설명

### Q6-1. 인증 감사 로그와 업무 감사 로그를 왜 분리했나요?

짧은 답변:
보안 사건과 비즈니스 상태 전이는 운영 목적이 달라서, 같은 테이블/화면보다 별도 audit trail로 분리하는 편이 분석과 권한 모델이 더 명확하다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- `auth_audit_log`: 로그인, refresh, 소셜 연결/해제
- `domain_audit_log`: 입학 waitlist/offer/승인, 출결 요청 승인/거절, 공지 수정/삭제
- principal이 보는 화면도 `/audit-logs`와 `/domain-audit-logs`로 분리했다고 설명

## 3. 인증/JWT

### Q7. JWT refresh token 설계를 어떻게 개선했나요?

짧은 답변:
처음의 이메일 단위 단일 refresh key 구조를 세션 단위 Redis 저장 구조로 바꾸고, refresh rotation과 revoke 전략을 넣었습니다.

꼬리 질문 대응 포인트:
- 단일 key는 멀티 디바이스에 약하다고 설명
- `sessionId` 기반으로 refresh token을 저장한다고 설명
- 로그아웃은 현재 세션 revoke, 탈퇴는 전체 세션 revoke라고 설명

### Q8. 왜 refresh rotation이 필요하다고 봤나요?

짧은 답변:
refresh token이 오래 살아 있는 인증 수단이기 때문에, 같은 값을 계속 재사용하게 두면 탈취 이후 대응이 약해지기 때문입니다.

꼬리 질문 대응 포인트:
- refresh 시 새 refresh token으로 교체
- 서버 저장값도 같이 갱신
- mismatch면 즉시 실패 처리

### Q8-1. 세션 종료 시 access token도 바로 끊어지게 한 이유는 무엇인가요?

짧은 답변:
refresh token만 지우면 이미 발급된 access token은 TTL 동안 계속 살아 있기 때문에, 활성 세션 관리가 실제 보안 제어가 되려면 access token도 세션 레지스트리에 묶여야 한다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- JWT 필터에서 `memberId + sessionId`로 Redis 세션 유효성을 추가 확인
- 세션 revoke 시 refresh token뿐 아니라 해당 access token 인증도 즉시 차단
- Redis 장애 시에는 가용성을 위해 fail-open으로 두고, 이 트레이드오프를 문서화했다고 설명

### Q9. 로그인/refresh rate limit은 어떻게 설계했나요?

짧은 답변:
Redis fixed-window 방식으로 로그인은 IP와 이메일 기준을 같이 보고, refresh는 IP 기준으로 제한했습니다.

꼬리 질문 대응 포인트:
- 로그인은 성공이 아니라 실패만 예산을 쓰도록 정교화했다고 설명
- refresh는 짧은 시간 내 반복 호출 남용 방지 목적이라고 설명
- 429와 비즈니스 코드로 일관되게 응답한다고 설명

### Q10. client IP는 왜 별도 resolver를 만들었나요?

짧은 답변:
`X-Forwarded-For`를 무조건 신뢰하면 헤더 스푸핑으로 rate limit을 우회할 수 있어서, trusted proxy일 때만 전달 헤더를 믿도록 분리했습니다.

꼬리 질문 대응 포인트:
- loopback과 설정된 proxy만 신뢰
- 그 외에는 `remoteAddr`만 사용
- 보안 기능은 네트워크 신뢰 모델까지 같이 봐야 한다고 설명

## 4. OAuth2 / 소셜 로그인

### Q11. OAuth2 계정 자동 연결을 왜 금지했나요?

짧은 답변:
같은 이메일이라고 바로 기존 계정에 소셜 로그인을 붙이면 계정 식별과 복구 정책이 모호해져서, 자동 연결보다 명시적 연결이 안전하다고 판단했습니다.

꼬리 질문 대응 포인트:
- 이메일 충돌 시 안내 후 기존 로그인 방식으로 유도
- settings 화면에서 사용자가 직접 link 시작
- 임시 OAuth 세션 정리까지 같이 처리

### Q12. 소셜 전용 계정에 로컬 비밀번호 설정 기능을 넣은 이유는 무엇인가요?

짧은 답변:
소셜 로그인만 가능한 계정은 인증 수단이 하나뿐이라 계정 잠금 리스크가 커서, 사용자가 로컬 로그인 수단을 확보할 수 있게 했습니다.

꼬리 질문 대응 포인트:
- password bootstrap endpoint
- 이미 로컬 비밀번호가 있으면 차단
- settings 화면에서 분기 UI 제공

### Q13. 같은 provider 소셜 계정 교체를 왜 막았나요?

짧은 답변:
Google/Kakao의 `providerId`를 비밀번호가 아니라 이메일과 같은 로그인 식별자로 봤기 때문입니다.

꼬리 질문 대응 포인트:
- unlink 후 다른 같은 provider 계정으로 재연결하는 흐름까지 차단
- historical binding을 남기기 위해 `unlinked_at` 이력 구조 도입
- 동일 providerId 재연결만 허용

### Q14. 소셜 연결 해제는 어떤 조건에서 허용하나요?

짧은 답변:
다른 로그인 수단이 남아 있을 때만 허용합니다. 즉 로컬 비밀번호가 있거나, 다른 소셜 provider가 연결돼 있어야 합니다.

꼬리 질문 대응 포인트:
- 마지막 로그인 수단을 잃어버리면 계정 잠금이 되므로 차단
- `A010`으로 명시적 실패 처리

## 5. 상태 전이 / 업무 감사

### Q14-1. 왜 승인 즉시 등록 대신 waitlist/offer 상태를 분리했나요?

짧은 답변:
반 정원과 운영 현실을 코드로 설명하려면 `승인/거절`만으로는 부족했고, 대기열과 제안 수락/만료가 있는 상태 전이가 더 맞다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- `WAITLISTED`, `OFFERED`, `OFFER_EXPIRED` 상태 설명
- 수락 시점에만 실제 `Kid` 생성
- 승인 하나로 끝나는 CRUD보다 운영형 워크플로우를 보여주기 좋다고 설명

### Q14-2. 정원 초과와 동시성은 어떻게 다뤘나요?

짧은 답변:
반 row를 잠그고, 현재 원생 수뿐 아니라 active offer까지 좌석 점유로 계산했습니다.

꼬리 질문 대응 포인트:
- `ClassroomCapacityService`에서 반 lock
- `Kid` 수 + `OFFERED` 신청 수를 합산
- 직접 원생 등록과 반 이동에도 같은 검증을 적용해 모델 일관성을 유지했다고 설명

### Q14-3. 출결 요청을 `Attendance`와 별도 aggregate로 분리한 이유는 무엇인가요?

짧은 답변:
학부모가 최종 출결 record를 직접 바꾸지 못하게 하고, 요청과 승인 사이에 review queue를 두기 위해서입니다.

꼬리 질문 대응 포인트:
- `AttendanceChangeRequest`는 요청/승인/거절/취소 이력 보존용
- 최종 `Attendance`는 승인 시점에만 upsert
- 권한 경계와 이력 보존을 동시에 만족한다고 설명

### Q14-4. 인증 감사 로그와 업무 감사 로그를 왜 분리했나요?

짧은 답변:
로그인/refresh 같은 보안 사건과 입학/출결/공지 상태 변경은 운영 목적이 달라서 같은 테이블에 섞지 않는 게 더 명확하다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- `auth_audit_log`는 보안 사건 분석용
- `domain_audit_log`는 비즈니스 상태 전이 추적용
- principal 조회 화면과 CSV export도 각각 역할에 맞게 분리했다고 설명

## 6. 테스트/CI

### Q15. 왜 H2 대신 Testcontainers를 썼나요?

짧은 답변:
이 프로젝트의 핵심 리스크가 MySQL/Flyway/Redis/JWT 세션 같은 운영 스택에 있었기 때문에, H2로는 그 리스크를 제대로 검증할 수 없다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- MySQL/Redis Testcontainers + Flyway로 전환
- refresh/session, soft delete, JPQL, 인덱스 전제 등을 실제 환경과 더 가깝게 검증
- "테스트가 있다"보다 "무엇을 검증하느냐"가 중요하다고 설명

### Q16. CI를 왜 fast/integration/performance smoke로 나눴나요?

짧은 답변:
개발 피드백 속도와 실환경형 검증을 같이 가져가기 위해서입니다.

꼬리 질문 대응 포인트:
- fast checks는 빠르게 깨지는 영역
- integration suite는 Testcontainers 기반 전체 흐름
- performance smoke는 audit console 같은 운영 경로의 query budget 회귀를 잡는 용도라고 설명
- path include가 아니라 JUnit suite로 분리해 어떤 테스트가 어느 job에 속하는지 명시적으로 관리한다고 설명

### Q17. 이번 프로젝트에서 테스트로 가장 인상 깊게 본 문제는 무엇이었나요?

짧은 답변:
`REQUIRES_NEW`로 저장한 감사 로그가 테스트 트랜잭션 스냅샷에서는 바로 안 보이는 문제가 있었고, assertion만 별도 트랜잭션으로 읽도록 고친 경험입니다.

꼬리 질문 대응 포인트:
- MySQL 격리 수준과 테스트 트랜잭션 가시성 문제 설명
- `readCommitted(...)` 헬퍼로 해결
- 테스트도 운영 동작을 이해해야 제대로 쓸 수 있다는 포인트

### Q17-1. 최근 추가한 운영형 워크플로우는 어떻게 테스트했나요?

짧은 답변:
입학 신청과 출결 요청 모두 상태 전이가 핵심이라, 성공 케이스만이 아니라 교차 유치원 차단, 정원 초과, 승인/거절, review queue 정렬까지 통합 테스트로 고정했습니다.

꼬리 질문 대응 포인트:
- `WAITLISTED`, `OFFERED`, `OFFER_EXPIRED` 상태 전이 검증
- `AttendanceChangeRequest`의 create/approve/reject/cancel 검증
- `Classroom.capacity` 초과 시 원생 생성/반 이동 실패 검증
- MySQL/Testcontainers 기준으로 검증했다고 설명

## 7. 성능

### Q18. 성능 최적화는 어떻게 접근했나요?

짧은 답변:
무작정 튜닝하지 않고, 재현 시나리오를 정의한 뒤 개선 전 쿼리 수와 응답 시간을 측정하고, 변경 후 같은 조건에서 다시 측정했습니다.

꼬리 질문 대응 포인트:
- Notepad `22 -> 4`, `15ms -> 4ms`
- Dashboard `13 -> 5`, `30ms -> 9ms`
- Dashboard 반복 조회 `5 -> 0`, `12ms -> 0ms`

### Q19. 가장 대표적인 성능 개선 사례 하나만 설명해 주세요.

짧은 답변:
알림장 목록에서 읽음 수를 row마다 따로 세던 N+1 패턴을 다건 집계 쿼리로 바꿔 쿼리 수를 크게 줄였습니다.

꼬리 질문 대응 포인트:
- 단건 count 반복 -> 한 번의 집계 조회
- 목록 성격 API는 개별 편의보다 배치 집계가 중요
- 성능 문서를 같이 남겼다고 설명

### Q20. 캐시는 어디에 적용했고 왜 적용했나요?

짧은 답변:
대시보드 통계처럼 짧은 시간 안에 반복 조회되지만 강한 실시간성이 꼭 필요하지 않은 영역에 60초 TTL 캐시를 적용했습니다.

꼬리 질문 대응 포인트:
- 반복 조회 비용이 큰 지표 API에 적합
- 캐시 적합성은 데이터 성격으로 판단했다고 설명

## 8. 운영 관측성 / 감사 추적

### Q21. Actuator는 왜 health/info만 공개했나요?

짧은 답변:
운영 체크에 필요한 최소 정보만 공개하고 나머지는 정보 노출을 줄이기 위해 닫는 게 맞다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- `/actuator/health`, `/actuator/health/readiness`, `/actuator/info`
- liveness/readiness 분리 설명
- `criticalDependencies`가 DB/Redis를 직접 probe 한다고 설명
- 장애 주입 테스트로 `readiness DOWN / liveness UP`을 검증했다고 설명

### Q21-1. Swagger/OpenAPI와 Prometheus는 운영에서 어떻게 다르게 다뤘나요?

짧은 답변:
local/demo에서는 시연과 개발 효율을 위해 열어 두고, prod에서는 Swagger/OpenAPI를 비활성화하고 management endpoint는 별도 포트로 분리해 노출면을 줄였습니다.

꼬리 질문 대응 포인트:
- `application-prod.yml`에서 `springdoc` 비활성화
- `management.server.port`, `management.server.address`로 운영 plane 분리
- audit console에서도 API 문서 링크를 설정값과 맞춰 조건부 렌더링한다고 설명

### Q21-2. 왜 학부모 출결 변경을 `Attendance` 직접 수정이 아니라 별도 aggregate로 분리했나요?

짧은 답변:
학부모 요청과 승인된 출결 사실은 성격이 달라서, 요청은 `AttendanceChangeRequest`, 확정 결과는 `Attendance`로 분리하는 편이 권한과 감사 추적 모두에 유리하다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- 승인 전에는 `Attendance`를 바꾸지 않음
- 교사/원장이 검토한 뒤 승인 시에만 `AttendanceService.upsertAttendance(...)` 호출
- 요청/승인/거절/취소를 모두 `domain_audit_log`에 남길 수 있다고 설명

### Q21-3. 입학 신청에 waitlist/offer 상태를 넣은 이유는 무엇인가요?

짧은 답변:
실제 운영에서는 좌석 제약 때문에 승인/거절 두 단계만으로는 흐름을 설명하기 어렵고, waitlist와 offer를 넣어야 상태 전이와 좌석 예약을 제대로 다룰 수 있기 때문입니다.

꼬리 질문 대응 포인트:
- `preferredClassroom`과 `assignedClassroom`을 분리한 이유 설명
- `offer`도 좌석 예약으로 계산해서 초과 제안을 막았다고 설명
- 만료 배치는 현재 단일 인스턴스 scheduler 기준으로 구현했고, 멀티 인스턴스는 확장 포인트라고 설명

### Q22. correlation id를 넣은 이유는 무엇인가요?

짧은 답변:
브라우저 요청, 서버 로그, 프록시 로그를 같은 요청 단위로 묶어 추적하기 위해서입니다.

꼬리 질문 대응 포인트:
- 요청이 헤더를 보내면 재사용
- 없으면 서버 생성
- 응답 헤더에도 다시 내려줌

### Q23. 인증 감사 로그 조회 API는 왜 principal만 보게 했나요?

짧은 답변:
인증 이벤트는 보안 민감도가 높아서 운영 책임이 큰 역할까지만 열어야 한다고 봤기 때문입니다.

꼬리 질문 대응 포인트:
- teacher/parent는 조회 불가
- 같은 유치원에 안전하게 귀속된 로그만 조회
- known email 실패는 write-time `kindergartenId`로 포함하고, 완전히 익명인 실패만 제외

### Q23-1. 외부 알림 전달을 왜 notification outbox로 분리했나요?

짧은 답변:
앱 내 알림 저장과 외부 webhook/email 전송을 한 트랜잭션 책임으로 묶으면, 외부 채널 장애가 핵심 요청 흐름을 흔들 수 있어서 분리했습니다.

꼬리 질문 대응 포인트:
- `notification` row 저장은 그대로 즉시 완료
- 외부 채널은 `notification_outbox`에 적재 후 worker가 비동기 전달
- 실패는 로그만 남기지 않고 retry/dead-letter 상태로 관리
- 현재는 단일 인스턴스 기준 claim 전략이고, 멀티 인스턴스 lock은 다음 단계 확장 포인트라고 설명

### Q23-2. 인증 이상 징후 감지, in-app alert, external incident delivery는 어떻게 이어지나요?

짧은 답변:
로그인 실패는 먼저 `auth_audit_log`에 남기고, threshold를 넘으면 principal에게 `AUTH_ANOMALY_DETECTED` in-app 알림을 저장한 뒤, 같은 알림을 `notification_outbox`로 넘겨 incident webhook 전달까지 이어지게 했습니다.

꼬리 질문 대응 포인트:
- principal은 `/audit-logs`와 CSV export로 사건을 바로 확인
- external webhook은 retry/dead-letter 상태로 추적
- Grafana는 auth event 지표를 보는 용도이고, outbox 전용 운영 콘솔은 아직 없다고 설명

## 9. 설계 트레이드오프 / 회고

### Q24. 가장 아쉬운 점은 무엇인가요?

짧은 답변:
외부 incident webhook과 retry/dead-letter까지는 넣었지만, outbox claim은 아직 단일 인스턴스 기준이라 멀티 인스턴스 안전성까지는 더 보강할 여지가 있습니다.

꼬리 질문 대응 포인트:
- 지금은 조회 후 상태 전이 방식이라 cluster-safe lock이 아님
- 다음 단계는 `FOR UPDATE SKIP LOCKED` 또는 메시지 브로커 검토
- 운영 알림도 webhook에서 Alertmanager/Slack/Email 정책까지 확장 가능
- outbox 상태는 현재 DB와 통합 테스트로 검증하고 있으며, 전용 운영 콘솔은 아직 없다고 설명

### Q25. 다음에 더 개선한다면 무엇을 하겠나요?

짧은 답변:
지금은 인증 incident alerting과 outbox 신뢰성까지 닫았기 때문에, 다음 단계라면 waitlist/출결 승인 같은 상태 머신과 업무 감사 로그를 붙이겠습니다.

꼬리 질문 대응 포인트:
- 반 정원, 입학 waitlist, offer expiry 같은 상태 전이 설계
- 학부모 출결 요청 -> 교사 승인 workflow
- `domain_audit_log`로 비즈니스 상태 변경 증적 저장

### Q26. 이 프로젝트를 통해 무엇을 배웠나요?

짧은 답변:
기능을 만드는 것보다, 기능이 많아질수록 권한 경계와 인증 lifecycle, 테스트 현실성, 운영 추적이 더 중요해진다는 점을 배웠습니다.

꼬리 질문 대응 포인트:
- 백엔드는 "동작"보다 "안전하게 운영 가능한가"가 더 중요하다고 설명
- 포트폴리오도 기능 나열보다 설계 판단과 트레이드오프로 보여줘야 한다고 설명
