# Phase 25: 로그인 Rate Limit 정책 정교화

## 배경

- 이전 rate limit 배치는 로그인과 refresh API의 남용 방어 자체를 추가하는 데 초점이 있었다.
- 하지만 로그인 정책을 다시 보면, 정상 로그인 성공도 이메일/IP 예산을 함께 소비하고 있었다.
- 이 상태에서는 사용자가 짧은 시간에 여러 번 정상 로그인하거나 테스트 중 반복 로그인할 때도 같은 실패 예산을 써버려 운영 UX가 불필요하게 나빠진다.

## 목표

1. 로그인 rate limit을 brute-force 방어에 더 가깝게 만든다.
2. 성공 로그인은 실패 예산을 소모하지 않도록 한다.
3. 성공 로그인 시 이메일 기준 실패 카운터를 초기화해 정상 사용자 잠금 가능성을 줄인다.

## 결정

1. 로그인 진입 시에는 현재 실패 횟수만 조회해서 제한 여부를 판단한다.
2. 실제 카운터 증가는 인증 실패 시점에만 수행한다.
3. 성공 로그인 시에는 이메일 기준 실패 카운터를 삭제한다.
4. IP 실패 카운터는 남겨둬서 짧은 시간 내 광범위한 실패 흔적은 유지한다.

## 구현 요약

### 1) AuthRateLimitService 분리

- `validateLoginAllowed`
  - 더 이상 증가시키지 않고 현재 값만 조회
- `recordLoginFailure`
  - 인증 실패 시 로그인 IP/이메일 카운터 증가
- `clearLoginFailures`
  - 성공 로그인 시 이메일 기준 실패 카운터 삭제

### 2) AuthService 흐름 정리

- 로그인 처리 순서:
  - 사전 제한 확인
  - 인증 시도
  - 실패 시 `recordLoginFailure`
  - 성공 시 `clearLoginFailures`

이렇게 바꿔서 "로그인 시도 제한"이 아니라 "로그인 실패 제한"에 더 가깝게 동작하도록 만들었다.

## 테스트

- `AuthApiIntegrationTest`
  - 동일 사용자의 반복 성공 로그인은 `429`로 막히지 않는지 검증
  - 일부 실패 후 성공 로그인 시 이메일 실패 카운터가 초기화되는지 검증
  - 기존 실패 누적 및 trusted proxy 기반 IP 해석 테스트도 유지

## 검증 결과

- `./gradlew compileJava compileTestJava`
- `./gradlew test --tests "com.erp.api.AuthApiIntegrationTest"`

둘 다 통과했다.

## 인터뷰 답변 포인트

### 왜 단순 횟수 제한에서 한 단계 더 손봤는가

- 보안 장치는 "막는다"만으로 끝나지 않고 정상 사용자 행동을 얼마나 덜 방해하는지도 중요하다.
- 실패와 성공을 같은 예산으로 묶으면 방어는 쉬워 보여도 실제 운영성은 떨어진다.

### 왜 이메일 카운터만 초기화했는가

- 성공 로그인 후에도 같은 IP에서 광범위한 실패가 있었던 흔적은 남길 필요가 있다.
- 그래서 사용자 잠금과 직접 연결되는 이메일 실패 카운터만 지우고, IP 실패 카운터는 유지하는 쪽을 선택했다.

### 트레이드오프는 무엇인가

- NAT 환경에서는 같은 공인 IP를 쓰는 여러 사용자의 실패 흔적이 일부 공유될 수 있다.
- 현재 단계에서는 단순성과 설명 가능성을 우선했고, 이후 필요하면 IP 외에 device/session fingerprint 같은 축을 추가할 수 있다.
