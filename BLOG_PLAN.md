# BLOG_PLAN.md

## 작업명
- 취업용 개발 블로그 시리즈 기획 1차 (연재 구조 + 초심자용 설명 전략 + 글별 파일/클래스 범위 정의)

## 현재 상태
- 진행 중: 블로그 연재 구조 설계
- 이번 턴 목표: `blog/` 폴더를 작업 공간으로 만들고, 블로그 작업 전용 SSOT를 별도로 분리한다.

## 1) 목표 / 범위
- 이 저장소 전체 개발 과정을 **Java / Spring Boot 입문자도 따라갈 수 있는 장기 개발 블로그 시리즈**로 재구성한다.
- 단순 회고가 아니라, “왜 이 순서로 설계했고 어떤 자바 클래스/메서드를 만들었는지”를 단계별로 설명할 수 있는 집필 계획을 만든다.
- 범위는 프로젝트 부트스트랩부터 현재 상태까지 전체다.
  - Gradle 설정
  - Docker / MySQL / Redis / profile 구성
  - 도메인 모델링
  - 인증 / JWT / OAuth2
  - 출석 / 알림장 / 공지 / 신청 / 승인
  - 멀티테넌시 보안 하드닝
  - Testcontainers / CI / Observability
  - 감사 로그 / outbox / 운영형 워크플로우
  - 포트폴리오 문서화 / 데모 / 면접 패키징
- 이번 단계에서는 실제 본문 전체를 쓰지 않고, **연재 계획과 집필 전략을 SSOT 문서로 먼저 고정**한다.

## 2) 세부 작업 단계
1. 블로그 작업용 SSOT 분리
   - 기존 개발 SSOT인 `PLAN.md`, `PROGRESS.md`는 유지
   - 블로그 작업 전용으로 `BLOG_PLAN.md`, `BLOG_PROGRESS.md` 사용

2. 작업 공간 구성
   - 루트 `blog/` 폴더 생성
   - `blog/README.md`에 폴더 역할과 집필 원칙 명시
   - `blog/_post_template.md`에 공통 글 구조 정의

3. 입력 자료 범위 고정
   - `README.md`
   - `docs/decisions/*`
   - `docs/portfolio/*`
   - `docs/guides/*`
   - `src/main/java`
   - `src/test/java`
   - `build.gradle`
   - `docker/*`
   - `application*.yml`

4. 독자 정의
   - Java / Spring Boot를 처음 공부하는 입문자
   - 첫 포트폴리오 프로젝트를 시작하는 취업 준비생
   - “무엇부터 설계해야 하는지” 감이 없는 독자

5. 글쓰기 원칙 정의
   - 기능 나열보다 설계 이유를 먼저 설명
   - 각 글마다 실제 파일/클래스/메서드 이름을 반드시 명시
   - 구조는 `문제 -> 설계 -> 코드 -> 테스트 -> 회고 -> 취업 포인트`
   - 입문자용 설명과 취업 포인트를 분리해 적는다
   - 결정 로그는 사실 근거, 블로그는 학습형 서사로 역할을 분리한다

6. 연재 구조 설계
   - 시리즈 전체를 4개 구간으로 나눈다
     - 기초 구축
     - 핵심 도메인 구현
     - 운영형 백엔드 고도화
     - 포트폴리오 패키징
   - 각 글마다 아래를 고정한다
     - 글 목적
     - 다룰 파일
     - 핵심 클래스 / 메서드
     - 입문자 설명 포인트
     - 취업 포인트

7. 산출물 작성
   - `blog/README.md`
   - `blog/00_series_plan.md`
   - `blog/_post_template.md`
   - 이후 실제 글은 `blog/01-*.md` 형식으로 순차 작성

8. 집필 우선순위 고정
   - 먼저 써야 하는 글
     - Gradle / Spring Boot bootstrap
     - Docker / MySQL / Redis 환경 구성
     - 첫 Security / JWT 설계
     - 멀티테넌시 권한 하드닝
   - 이유
     - 입문자 유입이 많고
     - 포트폴리오 차별점도 빠르게 드러나기 때문

## 3) 검증 계획
- 루트에 `BLOG_PLAN.md`, `BLOG_PROGRESS.md`가 있어야 한다.
- `blog/` 폴더가 생성되어 있어야 한다.
- 계획 문서에 아래가 모두 포함되어야 한다.
  - 전체 연재 개요
  - 글별 순서
  - 글별 핵심 파일 / 클래스 / 메서드
  - 초심자용 설명 포인트
  - 실제 집필 우선순위
- 문서는 Markdown 기준으로 읽기 쉬워야 한다.
- `git diff --check` 통과

## 4) 리스크 및 대응
- 범위가 너무 커서 계획이 추상적으로 흐를 수 있음
  - 대응: 글마다 실제 파일 / 클래스 / 메서드를 적는다
- 초심자용 설명과 취업용 설명이 섞여 글이 장황해질 수 있음
  - 대응: 각 글에서 `입문자 설명`과 `취업 포인트`를 분리한다
- 기존 결정 로그와 블로그가 중복될 수 있음
  - 대응: 결정 로그는 근거 자료, 블로그는 재구성된 설명 자료로 역할을 나눈다
- 글 수가 많아지며 품질이 흔들릴 수 있음
  - 대응: `blog/_post_template.md`를 공통 템플릿으로 사용한다
