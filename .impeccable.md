# Kindergarten ERP Impeccable Context

이 파일은 Impeccable 기반 디자인/프론트엔드 개선의 repo-local 컨텍스트입니다.
전역 Codex, Claude, npm 설정에 의존하지 말고 이 저장소 안의 `.agents/skills/*`, `.npmrc`, `package.json`, `scripts/impeccable-detect.mjs`만 사용합니다.

## Design Context

### Users

- 원장(PRINCIPAL): 유치원 운영 현황, 승인 대기, 감사 로그, CSV export, 통계 지표를 빠르게 확인해야 하는 운영 관리자.
- 교사(TEACHER): 수업/등하원 흐름 중 출석, 알림장, 일정, 신청 상태를 짧은 시간 안에 처리해야 하는 현장 사용자.
- 학부모(PARENT): 모바일 또는 짧은 여유 시간에 알림장, 출결 상태, 입학/출결 요청 결과를 확인하는 보호자.
- 사용 맥락은 업무형 ERP다. 화면은 신뢰감, 빠른 스캔, 실수 방지, 명확한 상태 전이를 우선한다.

### Brand Personality

- 3 words: 차분한, 신뢰감 있는, 빠른.
- "Simple is Best"가 제품 철학이다. 장식보다 업무 완료 속도와 상태 이해를 우선한다.
- 어린이/교육 도메인의 따뜻함은 유지하되, 과한 유아풍, 장난감 같은 색, 이모지 남발은 피한다.

### Aesthetic Direction

- 기본 방향은 white-first 운영 도구다. 페이지 배경은 흰색 계열을 기본으로 유지하고, 녹색은 브랜드 accent와 상태/action에 제한적으로 사용한다.
- 정보 밀도는 중간 이상으로 유지한다. 원장/교사용 화면은 카드 장식보다 표, 필터, 상태 뱃지, 명확한 액션 배치를 우선한다.
- 학부모용 화면은 긴 설명보다 현재 상태, 다음 행동, 결과 피드백을 먼저 보여준다.
- 현재 구현은 Thymeleaf + Tailwind CDN + HTMX + Alpine.js이며, 공통 CSS는 `src/main/resources/static/css/custom.css`, 공통 JS는 `src/main/resources/static/js/app.js`에 있다.
- 기존 화면에는 반복되는 `rounded-2xl`, card-heavy layout, per-page Tailwind config, gradient primary button, decorative blurred blobs가 있다. 향후 개선은 대형 리팩터링보다 공통 token/component 정리와 화면별 polish를 점진 적용한다.

### Design Principles

1. 업무 상태가 먼저 보이게 한다: 승인 대기, 오류, 마감, 출석률, 감사 이벤트는 색만이 아니라 텍스트/아이콘/위치로도 구분한다.
2. 액션 위계를 명확히 한다: primary action은 화면당 적게 유지하고, 보조/파괴/필터 액션은 시각적으로 분리한다.
3. 카드 남용을 줄인다: 표/필터/요약 영역은 필요한 경우만 surface를 쓰고, nested card는 만들지 않는다.
4. 모바일에서 기능을 숨기지 않는다: 교사/학부모 핵심 액션은 44px 이상 touch target과 명확한 focus/active 상태를 가진다.
5. 포트폴리오 프로젝트답게 개선 전/후를 설명할 수 있게 한다: 화면 개선은 가능하면 detector 결과, 접근성/반응형 확인, 변경 의도를 함께 남긴다.

## Frontend Structure

- Templates: `src/main/resources/templates/**`
- Shared styles: `src/main/resources/static/css/custom.css`
- Shared UI helpers: `src/main/resources/static/js/app.js`
- Brand assets: `src/main/resources/static/img/logo.svg`, `logo.png`, `logo-mark.png`
- Frontend libraries are CDN-based: Tailwind CSS, HTMX, Alpine.js, SweetAlert2.

## Impeccable Workflow

1. 새 화면을 만들거나 큰 UI 변경 전: `$critique` 또는 `$audit` 성격으로 현재 화면의 문제를 먼저 정리한다.
2. 구현 중: `.impeccable.md`와 `.agents/skills/impeccable/SKILL.md`를 함께 읽고, 이 프로젝트의 light-first ERP 방향을 유지한다.
3. 완료 전: `$polish` 성격으로 spacing, hierarchy, copy, interaction states, responsive behavior를 점검한다.
4. 자동 detector: `npm run impeccable:detect`를 실행한다. 빠른 정규식 스캔은 `npm run impeccable:detect -- --fast`를 사용한다.
5. Spring/Thymeleaf 검증은 기존 Gradle 명령을 우선한다: `./gradlew compileJava compileTestJava`, 필요 시 `./gradlew check`.

## Avoid

- Purple/blue AI gradients, cyan-on-dark palettes, neon glow, dark glassmorphism.
- Gradient text, side-stripe accent borders, bounce/elastic easing.
- Decorative blurred blobs/orbs as a default background pattern.
- Same-sized card grids with generic icon + heading + text repeated across operational screens.
- Cards inside cards, excessive `rounded-2xl`/`rounded-3xl`, generic shadows as the main design idea.
- Emojis as primary business icons in principal/teacher workflows.
- Hardcoded colors or duplicated Tailwind config in every template when a shared token would work.
- Text smaller than 14px on mobile, weak focus states, color-only status communication.

## Current Design Debt To Track

- Per-template Tailwind config and inline `.bg-gradient-primary` definitions are duplicated.
- `custom.css` overrides many Tailwind gray/white classes globally, which can make future color reasoning harder.
- 페이지 배경에는 green-tinted wash, radial/blurred background, 장식 gradient를 사용하지 않는다.
- Several operational screens are card-heavy. Prefer table/filter/status layouts where they scan better.
- `app.js` still has development console logging for HTMX and SweetAlert confirm debug output; future UI polish should decide whether those logs remain local-only.
